Chapter 1

The Software Development Lifecycle

The SDLC is the end-to-end process of taking an idea and turning it into running software. Your team follows this every single month.

SQL analogy

Think of the SDLC like building a report. You start with a request (requirements), design the query logic (design), write the SQL (development), test it on sample data (testing), and schedule it to run in production (deployment). Software dev is the same — just bigger, with more people.

The six phases — mapped to your world

1
Planning / Requirements
A business team fills out your Confluence intake form with the control they want automated. This is requirements gathering. The output is a clear, agreed-upon scope before anyone writes a line of code.
2
Design
Your team decides how to build it — which microservice handles data movement, whether you use a direct Snowflake query or an API, what the R Markdown workpaper structure looks like.
3
Development
Developers write code in their feature branches. Jira tickets for data movement and for the R Markdown workpaper are the units of work. Each ticket = one feature branch in Bitbucket.
4
Testing
Code is tested — unit tests check individual functions, integration tests check that data flows correctly end-to-end. Your Jenkins pipeline often runs these automatically.
5
Deployment
Once tested, code is merged to main and Jenkins deploys it. Your team does this monthly in scheduled releases tied to Jira release versions.
6
Maintenance
After deployment, jobs run on schedule via Control-M. Bugs are filed as new Jira tickets. The cycle repeats next month.

Your monthly cadence fits the SDLC exactly

WeekWhat your team is doingSDLC phase
Week 1-2Working Jira tickets, writing code in feature branchesDevelopment
Week 2-3Code review, testing, fixing bugs found in reviewTesting
Week 3-4Merging to main, Jenkins runs, deploy to productionDeployment
OngoingControl-M runs jobs, monitoring, new intake forms arriveMaintenance → Planning
Chapter 2

Agile & Your Jira World

Agile is a philosophy of building software in short cycles with frequent feedback. Jira is the tool you use to run it.

What "Agile" actually means

Instead of planning everything upfront for 6 months and delivering once, Agile teams work in short cycles called sprints. At the end of each sprint, something working is delivered, reviewed, and adjusted.

Your context

Your team's monthly release cycle is essentially a 4-week sprint. Intake → ticket creation → development → review → deploy. Each release is a sprint with a hard monthly deadline.

Chapter 3

Git: The Basics

Git is version control — it tracks every change to every file, by every developer, with a full history. Bitbucket is where your team stores Git repositories.

R analogy

Think of Git like R Markdown with version history on steroids. Instead of saving workpaper_v2_FINAL_real_final.Rmd, Git tracks every change automatically with a message explaining what changed and why. You can always go back to any previous state.

Core concepts

TermWhat it meansReal-world analogy
repositoryA folder tracked by Git — all your code + full historyAn R project folder, but every save is logged forever
commitA saved snapshot of changes with a messageClicking "Save" on an R script, but labeled
branchAn isolated copy of the code to work inA duplicate of your Rmd file to experiment safely
mergeCombining changes from one branch into anotherCopying your tested changes back into the main file
pushUploading your local commits to BitbucketSaving your R script to SharePoint for teammates
pullDownloading the latest code from BitbucketRefreshing SharePoint to get a teammate's latest file
cloneCopying a full repo from Bitbucket to your machineDownloading a project folder from SharePoint for the first time

The daily Git workflow

# 1. Start work on a new Jira ticket (e.g., AUDIT-142) git checkout -b feature/AUDIT-142-payment-controls-pipeline # 2. Write code, then save a snapshot git add . git commit -m "AUDIT-142: add Snowflake query for loan balances" # 3. Upload to Bitbucket so team can see it git push origin feature/AUDIT-142-payment-controls-pipeline # 4. Open a Pull Request in Bitbucket for code review
Key rule

Developers should never commit directly to main. All work happens on a feature branch, gets reviewed via a Pull Request, then gets merged. This protects production code.

Chapter 4

Branching Strategy

Branches let multiple developers work in parallel without stepping on each other. Your team likely uses a strategy called GitFlow or a simplified version of it.

Your branch hierarchy

# main — production-ready code, deployed monthly main └── develop (integration branch — all features merge here first) ├── feature/AUDIT-142-payment-controls ├── feature/AUDIT-155-workpaper-rmd-update └── feature/AUDIT-161-snowflake-api-connector └── hotfix/fix-loan-balance-null-crash (emergency → main directly)

Branch naming conventions

# Feature work (tied to a Jira ticket) feature/AUDIT-142-payment-controls-pipeline # Bug fix bugfix/AUDIT-163-null-loan-id-crash # Emergency production fix hotfix/AUDIT-170-jenkins-job-timeout # Release preparation release/2024-07

The merge flow

1
Developer creates a feature branch from develop
Work is isolated. Other developers continue unaffected on their own branches.
2
Feature complete → Pull Request opened to develop
Code review happens here. You review, leave comments, approve or request changes.
3
All features merged to develop
Integration testing happens on develop — do all the pieces work together?
4
Release branch cut from develop
A release/2024-07 branch is created. Only bug fixes go here — no new features.
5
Release merged to main → Jenkins deploys
The monthly deployment fires. Jenkins picks up the merge to main and runs the pipeline.
Manager tip

Enforce branch naming in your team. Every branch should have a Jira ticket number in its name so you can always connect code changes to business requirements. "testing" or "gabe-changes-2" are not acceptable branch names.

Chapter 5

Pull Requests & Code Review

A Pull Request (PR) is a formal request to merge one branch into another. It's also the primary place where code quality is enforced.

Audit analogy

A Pull Request is like a workpaper review. The developer is the preparer. You (or a peer) are the reviewer. The PR stays open until the reviewer is satisfied. This is essentially the same QA process your audit team does on workpapers — just for code.

Anatomy of a good Pull Request

Chapter 6

CI/CD Concepts

CI/CD stands for Continuous Integration and Continuous Deployment. It's the automated system that takes code from Bitbucket and gets it running in production.

The warehouse analogy

Imagine a factory assembly line. A developer drops a part (code) onto the conveyor belt (CI/CD). It gets automatically inspected (tests), packaged (build), and shipped to the store shelf (deployed to production). No human carries it manually at each step.

CI — Continuous Integration

1
Code pushed to Bitbucket
Developer runs git push. Bitbucket triggers a webhook — it pings Jenkins saying "new code arrived."
2
Jenkins picks it up
A Jenkins job starts automatically. It checks out the code, installs dependencies, and runs automated tests.
3
Pass or fail feedback
If tests pass — green light, the PR can be merged. If tests fail — the developer fixes it before merging.

CD — Continuous Deployment

Jenkins + Control-M

Your team's version

Your team does this monthly rather than continuously. That's called scheduled releases — still CI/CD, just with a human-controlled deployment gate. Many audit and compliance teams prefer this because change control requirements demand documented release windows.

Repeatability
The same deployment process runs every time. No "it worked on my machine" problems.
Audit trail
Every deployment is logged in Jenkins — who triggered it, what version, what time.
Rollback
If a deployment breaks something, Jenkins can redeploy the previous version from the last good commit.
Chapter 7

Jenkins in Your Stack

Jenkins is the automation engine your team uses to build, test, and deploy code. It also schedules your monthly jobs.

🔗
Triggered by Bitbucket
When code is pushed or merged, Bitbucket sends a signal to Jenkins to start a job automatically
📋
Runs your Jenkinsfile
Each repo has a Jenkinsfile — a script defining exactly what steps Jenkins runs for that project
📅
Schedules jobs
Like SQL Server Agent or Power Automate schedules, Jenkins runs jobs on a cron schedule
📊
Logs everything
Every run has a full log — you can see exactly what ran, what failed, and why

Reading a Jenkinsfile

pipeline { agent any stages { stage('Checkout') { steps { git branch: 'main', url: 'https://bitbucket.org/yourteam/audit-controls.git' } } stage('Install Dependencies') { steps { sh 'pip install -r requirements.txt' } } stage('Run Tests') { steps { sh 'python -m pytest tests/' } } stage('Deploy Workpaper Job') { when { branch 'main' } steps { sh './deploy.sh' } } } post { failure { mail to: 'audit-team@company.com', subject: 'Pipeline Failed' } } }

Jenkins vocabulary

TermWhat it means
Job / PipelineA configured automated task — one per repo or deployment process
StageA named step in the pipeline — Checkout, Test, Deploy, etc.
BuildOne execution of a pipeline — each run gets a number (#1, #2, etc.)
Build logThe full console output of a run — where you go when something fails
Cron triggerA schedule like 0 2 * * 1 meaning "every Monday at 2am"
WebhookA signal from Bitbucket to Jenkins — "new code, start a build"
WorkspaceThe folder on Jenkins' server where it runs your code
Manager tip

When a developer says "the build failed," your first question should be: "What stage did it fail at?" Checkout? Install? Tests? Deploy? The stage name and the build log will tell you 90% of what you need to diagnose the problem.

Chapter 8

Releases & Deployments

A release is a versioned, planned deployment of code. Your monthly cycle is a release cycle — here's how all the pieces connect.

1
Jira Release Version created
In Jira, create a Release (e.g., "2024-07 Monthly Deployment"). All tickets for this month are attached to it. This is your manifest of what ships.
2
All feature branches merged to develop
Developers finish their tickets, PRs are reviewed and approved, features land in develop. Integration testing happens here.
3
Release branch cut from develop
release/2024-07 is created. No new features. Only release-blocking bug fixes can go in now.
4
Release merged to main + tagged
The release branch merges to main. A Git tag is created: v2024.07.0. This is the permanent record of exactly what code is in production.
5
Jenkins deployment pipeline fires
The merge to main triggers Jenkins. Jenkins builds, tests, and deploys. Control-M picks up the new job definitions.
6
Jira tickets closed, release marked Done
Jira release version is marked as Released. Stakeholders are notified. The cycle starts again for next month.

Git tags — your permanent version markers

# Create a tag after merging to main git tag -a v2024.07.0 -m "July 2024 monthly release" git push origin v2024.07.0 # To see all tags (your release history) git tag --list # To go back to a specific release (read-only) git checkout v2024.06.0
Audit relevance

Tags are your evidence. When an auditor asks "what version of the payment controls workpaper was running in Q2?", you point to the Git tag. Every release is permanently preserved and can be recreated exactly.

Chapter 9

Code Review Guide for Managers

As a manager you'll review Pull Requests. You don't need to catch every bug — you need to ensure quality, consistency, and maintainability.

Workpaper review analogy

You already know how to review a workpaper — you check that the logic is sound, the conclusion is supported, the format is consistent, and nothing is missing. Code review is the same process applied to scripts and pipelines instead of documents.

The code review checklist

CategoryWhat to checkRed flag example
PurposeDoes the code match the Jira ticket?PR for AUDIT-142 but the code changes the workpaper footer
ReadabilityCan you understand what the code does?def x(a, b) instead of def calculate_loan_balance(loan_id, date)
HardcodingAre credentials or paths hardcoded?password = "MyPass123" in the code
Error handlingWhat happens when Snowflake is down?No try/except block around database calls
TestsAre there tests for new logic?New SQL transformation with no test to verify output
Scope creepIs the change limited to the ticket scope?Bug fix ticket that also refactors three other functions

How to leave effective review comments

Be specific
"Line 42: this query will fail if loan_status is NULL — add WHERE loan_status IS NOT NULL" is helpful. "Needs work" is not.
Distinguish severity
Use prefixes: "blocker:" for must-fix before merge, "suggestion:" for improvements, "question:" if you just don't understand something
Explain the why
Don't just say change it — explain why. "This could cause a timeout on large loan volumes because..."
Approve when...

The code does what the ticket asks. No hardcoded credentials. Logic is understandable. Errors are handled. Scope is appropriate. Minor style issues can be noted as suggestions without blocking.

Request changes when...

There are security issues (credentials, exposed data). Logic is clearly wrong. No error handling on critical paths. The change breaks existing functionality. Scope is way beyond the ticket.
