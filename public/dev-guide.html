Chapter 1

The Software Development Lifecycle (SDLC)

The SDLC is the complete process of taking a business need and turning it into running, maintained software. Your team lives this every month.

The report-building analogy

Building a scheduled audit report and building a software feature follow almost identical steps. Someone requests it → you design the logic → you build it → you test it with sample data → you deploy it to run automatically → you maintain it. The SDLC is just this process formalized and scaled up.

Phase 1: Planning & Requirements

This is where work originates. The business comes to you with a control they want automated. Requirements gathering means getting all the details about what is needed — not yet worrying about how.

In your world this is the Confluence intake form. The line of business fills out the control description, the data sources involved, the frequency, and the expected output. A poorly filled intake form = a poorly scoped project = rework later. This is the most important phase because mistakes here compound through every phase that follows.

What good requirements look like

Good requirements are specific, testable, and agreed upon. "Automate the mortgage payment control" is bad. "Every first Monday of the month, extract all loans from the LOAN_PAYMENTS Snowflake table where payment_status = 'delinquent' and as_of_date = last month end, run them through the R Markdown workpaper, and deposit the output PDF into the GREAT MS3 repository" is good.

Phase 2: Design & Architecture

Once requirements are agreed on, the team decides how to build it. This is architecture. Which microservice handles what? Does the data come via API call or direct Snowflake query? What does the R Markdown workpaper structure look like? Where do secrets (passwords, tokens) get stored securely?

Design decisions made here are expensive to reverse. A bad database schema, a wrong API choice, a poorly structured microservice — these become technical debt that slows your team down for months or years.

Phase 3: Development

Developers write the actual code. Each Jira ticket represents one unit of work. Each unit of work gets its own Git branch. Developers work in isolation on their branch, commit often, and push to Bitbucket so their work is backed up and visible.

Good development includes writing the code and writing tests for it. Tests are code that checks your code works. Think of them like data validation rules — a test says "given this input, I expect this output" and fails loudly if it doesn't match.

Phase 4: Testing

There are several layers of testing. Understanding all of them matters because as manager you'll hear these terms in code reviews and Jenkins logs.

Test typeWhat it checksExample in your world
Unit testOne small function in isolationDoes calculate_delinquency_rate(loans) return the right number for a given list?
Integration testMultiple components working togetherDoes the Python script connect to Snowflake, run the query, and produce a DataFrame with the right columns?
Regression testThat old things still work after new changesAfter adding a new column to the workpaper, do all the old sections still render correctly?
End-to-end testThe full workflow from trigger to outputDoes the Control-M job trigger, extract from Snowflake, process in R, and produce the final PDF — all without errors?

Phase 5: Deployment

Deployment is moving tested, approved code from the development environment into production where it runs for real. Your team does this monthly. Jenkins automates the steps — it doesn't require a developer to manually copy files or run scripts on the production server.

A good deployment is boring. Nothing unexpected happens. The automation takes over and the new version is live. A bad deployment is exciting — and excitement in production is never good.

Phase 6: Maintenance & Monitoring

After deployment, Control-M runs your jobs on schedule. But things break. Snowflake table schemas change. APIs get updated. Business rules evolve. Maintenance is the ongoing work of keeping deployed software healthy. New bugs get filed as Jira tickets and the cycle begins again.

Manager insight

A huge percentage of real-world development work is maintenance — not greenfield builds. When your team is "slower than expected," they're often dealing with maintenance work that wasn't on the roadmap: a Snowflake view changed and broke two workpapers, a library was updated and broke a Python script, a Control-M job started timing out. Budget time for this.

Chapter 2

Agile & Your Jira World

Agile is both a philosophy and a set of practices for delivering software in short, iterative cycles. Jira is the operational tool that runs it day-to-day.

Waterfall vs Agile — why it matters

The old way (Waterfall) was: spend months gathering all requirements, then months designing, then months building, then months testing, then deploy once. The problem: by the time you deployed, the business had changed. You'd built the wrong thing.

Agile says: deliver something working every few weeks, get feedback, adjust, repeat. Each cycle is called a sprint. Shorter feedback loops mean fewer wrong turns.

Your team's version

Your monthly release cadence is Agile-style. Intake → design → build → review → deploy, all within 4 weeks. The hard monthly deadline functions like a sprint deadline. You're already doing it — you just may not have had the vocabulary for it.

The Jira hierarchy — fully explained

Jira organizes work in a hierarchy. Understanding this helps you structure tickets correctly so the team knows exactly what to work on and how it ladders up to business goals.

Epic
A large business objective or theme of work
Think of it as a chapter in a book. It contains many stories and may span multiple monthly releases.
Example: "Q3 Mortgage Servicing Controls Automation"
Story
A user-facing feature written from the user's perspective
Format: "As a [user], I want [feature] so that [business value]." Forces the team to think about why they're building something.
Example: "As an auditor, I want the payment delinquency control to run automatically so I don't have to manually extract data each month"
Task
A specific technical unit of work — one developer, one Jira ticket, one Git branch
This is what a developer actually works on day-to-day. Concrete, estimable, completable in days not weeks.
Example: "Build Snowflake extraction query for delinquent loans" or "Create R Markdown workpaper template for payment controls"
Bug
Something that is broken and needs fixing
Bugs go through the same workflow as tasks. They have priority levels — a bug that crashes production workpapers is P1 (fix immediately), a cosmetic formatting issue is P3 (fix when convenient).
Example: "Workpaper crashes when loan_count exceeds 50,000 rows"
Subtask
A checklist item inside a task — too small for its own ticket but worth tracking
If a task has 3-4 discrete steps, subtasks track each step's completion.
Example: Under "Build Snowflake extraction": "Write query", "Add error handling", "Write unit test", "Get peer review"

Sprint ceremonies — what actually happens in each meeting

Sprint Planning
At the start of the month: select tickets from backlog for this sprint. Estimate effort. Assign to developers. Output: a committed sprint backlog.
Daily Standup
15-minute daily check-in. Each person: what did I do yesterday, what am I doing today, any blockers? Not a status report — it's a blocker-clearing meeting.
Sprint Review
End of sprint: demo what was built to stakeholders. Get feedback. Did we build the right thing?
Retrospective
Team-internal: what went well, what didn't, what do we change next sprint? The most valuable meeting for continuous improvement.

Story points — estimating work

Story points are a relative measure of effort — not hours. A 1-point task is trivially easy. An 8-point task is complex with many unknowns. Teams use the Fibonacci sequence (1, 2, 3, 5, 8, 13) intentionally — the gaps between large numbers reflect increasing uncertainty.

Common mistake

New managers often try to convert story points to hours. Don't. A task that's 5 points for a senior developer might be 8 points for a junior developer. Story points measure complexity relative to the team, not absolute time. Use team velocity (average points completed per sprint) to predict delivery, not individual point-to-hour conversion.

Chapter 3

Git: Core Concepts

Git is a distributed version control system. It tracks every change to every file, who made it, when, and why. Bitbucket is the hosted server where your team's Git repositories live.

The R workpaper analogy

Imagine if R Studio automatically saved every version of every Rmd file you ever worked on, tagged each save with your name, timestamp, and a note about what you changed — and let you instantly compare any two versions or restore any previous state. That's Git. No more workpaper_v3_FINAL_actual_final_2.Rmd.

The fundamental Git concepts

repository
A project folder that Git is tracking — contains all code AND the complete history of every change ever made
Like an R project folder, except every file save since day one is preserved forever. You can travel back to any point in time.
Your team has one repo per microservice: MS1 repo, MS2 repo, MS3 repo, MS4 repo, plus repos for R Markdown workpapers and Python notebooks.
commit
A permanent snapshot of all staged changes at a point in time, with a message explaining what changed
Like pressing Save in R, but Git forces you to write a note explaining why you saved. Each commit gets a unique ID (a hash like a3f9c2d).
"AUDIT-142: add NULL check for loan_status in payment query" — a good commit message. "fixed stuff" — a bad one.
staging area
A holding area where you collect changes before committing them — you choose exactly what goes into each commit
Like a drafts folder. You make changes to 5 files, but you only stage 3 of them for this commit. The other 2 will go in the next commit.
git add snowflake_query.py stages just that file. git add . stages everything changed.
branch
A parallel, isolated version of the codebase where you can make changes without affecting anyone else's work
Like having your own personal copy of the entire codebase that's siloed from your teammates. You can break things, experiment, and rebuild — it doesn't touch their work.
Developer A works on feature/AUDIT-142 while Developer B works on feature/AUDIT-155. Neither sees the other's changes until merge.
merge
Combining the commit history from one branch into another — bringing work together
Like copying your tested R script changes into the shared team version of the file. Git does it automatically and handles conflicts.
Feature branch → develop branch merge happens via Pull Request. Develop → main merge happens at monthly release.
push
Uploading your local commits to the remote repository on Bitbucket so others can see your work
Like saving your R script to the shared SharePoint folder so teammates can see it. Until you push, your work only exists on your machine.
git push origin feature/AUDIT-142 — uploads your branch to Bitbucket.
pull
Downloading the latest commits from Bitbucket to your local machine — syncing up with what teammates have done
Like refreshing SharePoint to get your teammate's latest file version. Always pull before starting work each day.
git pull origin develop — get the latest develop branch before starting a new feature.
clone
Downloading a complete copy of a remote repo (including all history) to your local machine for the first time
Like downloading a whole project folder from SharePoint for the first time. You only clone once per repo — after that you use pull.
git clone https://bitbucket.org/yourteam/audit-ms1.git
remote
The version of the repository that lives on Bitbucket (as opposed to your local copy on your laptop)
The SharePoint version vs the copy on your local drive. Remote is the canonical, shared version. Your local copy is your workspace.
origin is the default name for the remote. git remote -v shows you where it points.
HEAD
A pointer that tells Git "this is the commit you're currently on" — your current position in history
Like a bookmark in a book. It marks where you are right now. When you switch branches, HEAD moves to point at that branch.
When Git says HEAD -> main it means you're currently on the main branch, at its latest commit.
Chapter 4

Git: Command Notebook

Your personal reference for frequently used Git commands — organized by situation so you can find what you need fast.

Getting oriented — commands you run to understand where you are

# What branch am I on? What files have I changed? git status # See recent commit history — one line per commit git log --oneline # See recent commits with more detail (author, date, full message) git log --oneline --graph --decorate --all # List all branches (local only) git branch # List all branches including remote Bitbucket branches git branch -a # Show what changed in a specific commit (use the short hash from git log) git show a3f9c2d # Show line-by-line diff of what you've changed but not yet staged git diff # Show diff of what's staged (ready to commit) vs last commit git diff --staged # Who last changed each line of a file? (great for blame/investigation) git blame snowflake_query.py

Daily development workflow

# Start your day — get the latest from Bitbucket git pull origin develop # Create a new branch for your Jira ticket git checkout -b feature/AUDIT-142-payment-controls # ... do your work, write code ... # See what you changed git status # Stage a specific file git add snowflake_query.py # Stage everything changed git add . # Commit with a descriptive message (Jira ticket ID first!) git commit -m "AUDIT-142: add NULL check for loan_status in extraction query" # Push your branch up to Bitbucket git push origin feature/AUDIT-142-payment-controls # If Bitbucket doesn't know about this branch yet (first push) git push --set-upstream origin feature/AUDIT-142-payment-controls

Switching between branches

# Switch to an existing branch git checkout develop # Modern syntax (Git 2.23+) — same thing git switch develop # Create a new branch AND switch to it in one step git checkout -b feature/AUDIT-155-new-workpaper # Checkout a remote branch that exists on Bitbucket but not locally yet git checkout -b feature/AUDIT-155 origin/feature/AUDIT-155

Undoing things — the most important section

# Unstage a file (remove from staging area, keep your changes) git restore --staged snowflake_query.py # Discard ALL changes to a file (permanently — goes back to last commit) git restore snowflake_query.py # Undo the last commit but KEEP your changes (staged and ready) git reset --soft HEAD~1 # Undo the last commit and UNSTAGE the changes (changes still in files) git reset HEAD~1 # SAFELY undo a commit by creating a new "reverse" commit (use on shared branches) git revert HEAD # Why revert vs reset? Reset rewrites history — dangerous on shared branches. # Revert adds a new commit that undoes the change — safe on shared branches.
Critical rule on shared branches

Never use git reset on main or develop — branches your teammates are also using. Reset rewrites Git history, which will corrupt your teammates' local copies. On shared branches, always use git revert instead. On your own personal feature branch, reset is fine.

Stashing — saving work temporarily

# You're mid-work and need to switch branches urgently # Stash saves your uncommitted changes so you can switch git stash # Switch to another branch, do emergency work, come back git checkout hotfix/urgent-fix # ... fix the thing ... git checkout feature/AUDIT-142 # Restore your stashed work git stash pop # See all stashes (you can have multiple) git stash list # Apply a specific stash without removing it from the list git stash apply stash@{0}

Working with Bitbucket remotes

# Fetch all changes from Bitbucket without merging them (safe to run anytime) git fetch origin # Pull = fetch + merge in one step git pull origin develop # See where your remote (origin) is pointing git remote -v # Delete a remote branch (after a PR is merged) git push origin --delete feature/AUDIT-142-payment-controls # Delete a local branch (after it's merged and you're done) git branch -d feature/AUDIT-142-payment-controls # Force delete a local branch (if it has unmerged commits — be careful) git branch -D feature/AUDIT-142-payment-controls

Tags — marking releases

# Create an annotated tag (use for releases — includes message) git tag -a v2024.07.0 -m "July 2024 monthly release" # Push the tag to Bitbucket git push origin v2024.07.0 # Push ALL local tags to Bitbucket at once git push origin --tags # List all tags git tag --list # See what commit a tag points to git show v2024.07.0 # Check out a specific release version (read-only — detached HEAD state) git checkout v2024.06.0

Investigating problems

# Find which commit introduced a bug using binary search git bisect start git bisect bad # current commit is broken git bisect good v2024.06.0 # this old version was fine # Git checks out the midpoint — test it, then tell Git: git bisect good # or: git bisect bad # Repeat until Git identifies the exact breaking commit git bisect reset # return to HEAD when done # Search commit messages for a keyword git log --grep="AUDIT-142" # See all commits that changed a specific file git log --follow snowflake_query.py
Chapter 5

Branching Deep Dive

Branching is one of Git's most powerful features and the source of most developer confusion. This chapter explains every branching concept including the ones that trip up experienced developers.

What a branch actually is under the hood

A branch is not a copy of files. It's a lightweight pointer to a commit. When you create a new branch, Git just creates a new pointer. No files are duplicated. This is why creating branches in Git is nearly instant — unlike older systems that literally copied files.

The sticky note analogy

Imagine the full history of your codebase as a timeline of saves. A branch is just a sticky note attached to one of those saves that says "this is where this line of work lives." When you make a new commit, the sticky note moves forward to point at your new commit. main is just another sticky note. develop is another. You can have hundreds of them.

Your GitFlow branch structure

# Your team's branch hierarchy main ← production. Only touched during monthly release. Tagged with version. │ └── develop ← integration branch. All features land here first. Always deployable. │ ├── feature/AUDIT-142-payment-controls ← one developer's work ├── feature/AUDIT-155-workpaper-update ← another developer's work ├── feature/AUDIT-161-snowflake-connector ← another developer's work │ └── release/2024-07 ← cut near month end. Bug fixes only. Then → main. └── hotfix/AUDIT-170-null-crash ← emergency fix. Goes directly to main AND develop.

Merge — combining branches

A merge takes the commits from one branch and integrates them into another. Git looks at the two branches, finds where they diverged, and combines the changes.

# Merge feature branch into develop git checkout develop # switch to the target branch git pull origin develop # make sure you have the latest git merge feature/AUDIT-142 # bring in the feature branch commits git push origin develop # push the result to Bitbucket

In practice your team does merges via Pull Requests in Bitbucket — not from the command line. The PR is the review gate before the merge happens.

Merge conflicts — what they are and how to handle them

A merge conflict happens when two branches changed the same line(s) of the same file and Git can't automatically decide which version to keep. Git stops the merge and asks you to decide.

# Git marks the conflict in the file like this: <<<<<<< HEAD (your current branch — develop) result_df = snowflake_conn.execute(query, timeout=30) ======= result_df = snowflake_conn.execute(query, timeout=60) >>>>>>> feature/AUDIT-155 (the branch being merged in) # You edit the file to keep what you want (delete the markers): result_df = snowflake_conn.execute(query, timeout=60) # chose 60 # Then complete the merge: git add snowflake_query.py git commit -m "merge AUDIT-155: resolved timeout conflict"

Rebase — rewriting history for cleaner commits

This is the term that confuses developers. Rebase is an alternative to merge. Instead of combining the history of two branches into a merge commit, rebase re-applies your commits on top of another branch — making it look like you branched off more recently.

After merge
A - B - C (develop) \ \ D - E - F (merge commit) History shows the branches diverged. One merge commit added.
After rebase
A - B - C - D' - E' (develop) History is linear. Commits D and E replayed on top of C. Cleaner log.
Golden rule of rebase

Never rebase a branch that other people are working on or that has been pushed to a shared remote. Rebase rewrites commit hashes — it creates new commits that look like your old ones but have new IDs. Anyone else who has the old commits will have history that no longer matches. Use rebase only on your own private feature branches, before they're merged.

# Typical use: bring your feature branch up to date with develop # instead of creating a merge commit, replay your commits on top of latest develop git checkout feature/AUDIT-142 # be on your feature branch git fetch origin # get latest from Bitbucket git rebase origin/develop # replay your commits on top of latest develop # If conflicts occur during rebase: # resolve the conflict in the file, then: git add . git rebase --continue # If you want to abort and go back to before the rebase started: git rebase --abort

Cherry-pick — taking one specific commit from another branch

Cherry-pick applies a single commit from one branch onto your current branch. Useful when you need just one specific fix without merging the entire branch.

# Scenario: a developer fixed a critical bug in their feature branch. # You need that fix on main right now, but their feature isn't ready. git log feature/AUDIT-142 --oneline # find the commit hash you want # output: a3f9c2d AUDIT-142: fix null loan_id crash git checkout main git cherry-pick a3f9c2d # apply just that commit to main

Interactive rebase — cleaning up messy commits before a PR

Interactive rebase lets you edit, reorder, combine, or delete commits before they're merged. Developers use this to clean up "work in progress" commits into a clean, readable history.

# Squash the last 3 commits into one clean commit git rebase -i HEAD~3 # This opens an editor showing: pick a1b2c3 WIP: started payment query pick d4e5f6 still working on it pick g7h8i9 AUDIT-142: completed payment query with NULL checks # Change to: pick a1b2c3 WIP: started payment query squash d4e5f6 still working on it squash g7h8i9 AUDIT-142: completed payment query with NULL checks # Git combines them into one commit with a message you write

Fast-forward merge vs three-way merge

You'll see this in Bitbucket PR options. A fast-forward merge happens when the target branch hasn't moved since you branched off — Git just moves the pointer forward. No merge commit is created. A three-way merge (or "merge commit") happens when both branches have diverged — Git creates a new commit that ties them together. Most teams use "squash and merge" in Bitbucket PRs — this combines all feature commits into one clean commit on develop.

Pull Requests & Code Review

A Pull Request (PR) is a formal proposal to merge one branch into another. It's also the primary quality gate in your team's workflow.

The workpaper review analogy

A PR is identical in purpose to a workpaper review in audit. Developer = preparer. Reviewer = you. The work product can't be "filed" (merged) until the reviewer approves. Comments must be addressed. The trail of review and approval is preserved. You've been doing code review your whole career — you just called it workpaper review.

What a good PR looks like

ElementWhat it should containWhy it matters
Title"AUDIT-142: Add Snowflake payment controls pipeline"Instantly tells you what changed and which ticket it belongs to
DescriptionWhat changed, why, what to test, any known limitationsGives reviewer context — makes review faster and better
Linked ticketBitbucket auto-links when branch name contains Jira IDTraceability from business requirement to code change
Small scopeOne ticket = one PR. Not "everything I did this month."Smaller PRs get better reviews. Large PRs get rubber-stamped.
Passing CIJenkins build is green before asking for reviewDon't waste reviewer's time on code that doesn't even run

PR review strategies — what to actually look for

As a manager reviewing PRs, you're not expected to catch every edge case. Your job is to check correctness of approach, security, maintainability, and whether the work matches the ticket.

CheckWhat to look forRed flag
SecurityNo hardcoded credentials, tokens, or passwords anywhere in the diffpassword = "Snowflake123!" in any file
Error handlingDatabase calls, API calls, and file operations have try/except blocksCode that will crash silently if Snowflake returns nothing
NamingFunctions, variables, and files have descriptive namesdef process(x) — process what? What is x?
ScopePR only contains changes related to its ticketTicket is a bug fix but diff shows refactored, unrelated functions

Writing effective review comments

Blocker
Must be fixed before this PR can merge. Used for security issues, bugs, or logic errors that would break production.

"Blocker: password is hardcoded on line 42 — move to environment variable"
Suggestion
Improvement worth making but not required to merge. Developer's discretion.

"Suggestion: consider extracting this query into its own function for reusability"
Question
You don't understand something — ask before approving. It might be fine, or it might reveal a bug.

"Question: what happens here if snowflake returns 0 rows?"
Nit
Tiny style issues not worth blocking over. State it clearly so the developer doesn't feel obligated.

"Nit: extra blank line on line 87 — not a blocker"
Chapter 7

CI/CD: The Full Picture

CI/CD is the automated system that takes code from a developer's commit and gets it running in production — without anyone manually copying files or running scripts.

The audit production analogy

Imagine if every time you saved a new version of an R Markdown workpaper, it automatically: ran all the data validation checks, rendered a test output PDF, flagged any errors back to you, and — if everything passed — deployed the new version to the production schedule. That is CI/CD. The goal is to make deployment as boring and automatic as possible so developers can focus on writing code, not moving files around.

CI — Continuous Integration in detail

Continuous Integration means every developer's code changes are merged into a shared branch frequently (at least daily) and automatically tested. The problem it solves: without CI, developers work in isolation for weeks, then try to merge everything at once — "merge hell."

1
Developer pushes code to Bitbucket
The moment git push runs, Bitbucket detects new commits on a branch. It immediately fires a webhook — an HTTP notification — to Jenkins saying "new code, start a build."
2
Jenkins starts a build automatically
Jenkins picks up the webhook, creates a fresh workspace, checks out the code from Bitbucket, and starts executing the Jenkinsfile stages in order. This happens with no human involvement.
3
Dependencies are installed
Jenkins runs pip install or installs R packages. Every build starts from a clean slate — no relying on "it worked on my machine."
4
Automated tests run
Unit tests, integration tests, and linting checks run. Jenkins tracks which tests pass and which fail. A single failing test fails the entire build.
5
Pass or fail result reported back
Jenkins updates the PR in Bitbucket with a green checkmark (pass) or red X (fail). If fail, the PR cannot be merged until the developer fixes the issue and pushes again — triggering a new build.

CD — Continuous Deployment in detail

After code passes CI and gets merged to main, the deployment pipeline fires. This is the "last mile" — getting code from the repository onto the server where it actually runs.

The key concepts you'll hear

webhook
An automatic HTTP notification from one system to another triggered by an event
Like an email notification except it's machine-to-machine. Bitbucket's webhook says to Jenkins: "event type: push, branch: feature/AUDIT-142, commit: a3f9c2d."
build artifact
The packaged output of a build — the thing that gets deployed, not the raw source code
Like the final rendered PDF from R Markdown. The source is the .Rmd file. The artifact is the PDF. For Python, the artifact might be a Docker container or a zip of the compiled code.
pipeline as code
The deployment pipeline steps are defined in a file in the repo (Jenkinsfile), not configured via a GUI
Instead of clicking through a UI to configure "run tests then deploy," you write a Jenkinsfile that lives in the repo with the code. The pipeline configuration is versioned just like the code.
rollback
Reverting a deployment to a previous known-good version
Like restoring a previous version of a SharePoint document after a bad edit. In Jenkins, you trigger a deployment of the previous Git tag (e.g., v2024.06.0 instead of v2024.07.0).
smoke test
A quick, minimal check after deployment to confirm the system is basically running
Like opening a workpaper after deploying to just check it renders and doesn't error before calling the deployment done. Not a full test — just "did it turn on?"
deployment gate
A manual approval step required before a pipeline continues to the next stage
Like requiring a manager sign-off before a workpaper is filed. The pipeline pauses, sends a notification, and waits for a human to click "approve" before deploying to production.
Chapter 8

Jenkins Deep Dive

Jenkins is your team's CI/CD engine. It's the thing that responds to code pushes, runs tests, and deploys to production. Understanding it deeply makes you a much better manager.

How Jenkins is structured

🖥️
Jenkins Server
A server running the Jenkins application. Your team has one (or more). It's always running, watching for webhooks and scheduled triggers.
📋
Jobs / Pipelines
A configured unit of work in Jenkins. Your team likely has one pipeline per repo (MS1, MS2, R workpapers, Python notebooks, etc.)
🤖
Agents / Nodes
The machines that actually run the pipeline steps. The Jenkins server orchestrates; agents do the work. Can be the same machine or separate.
📦
Workspace
A temporary folder on the agent where Jenkins checks out the code for each build. Deleted or overwritten each run for a clean start.

Anatomy of a Jenkinsfile — fully annotated

Manager debugging tip

When a build fails, search the console log for "ERROR", "FAILED", or "exit code". These are the actual failure points. Everything above them is context. The stage name tells you which phase failed. Releases & Deployments

A release is a carefully managed, versioned deployment of code changes. Your monthly release cycle is a textbook example of a scheduled release process.

Semantic versioning — how version numbers work

Version numbers follow a standard called Semantic Versioning (SemVer): MAJOR.MINOR.PATCH — for example v2024.07.0.

PartWhen it incrementsExample
MAJORBreaking changes — things that require existing users to change how they use the systemChanging a Snowflake view schema that all workpapers depend on
MINORNew features added in a backward-compatible wayNew workpaper added to the system without changing existing ones
PATCHBug fixes — no new features, nothing breakingFixing a calculation error in an existing workpaper

The complete monthly release playbook

1
Create Jira Release version (Day 1 of sprint)
In Jira: Releases → Create Release → "2024-07 Monthly Deployment". Set the target date. All tickets for this sprint get assigned to this release version. This is your shipping manifest.
2
Development in feature branches (Days 1–18)
Developers work their tickets. Each ticket gets a feature branch. PRs are opened, reviewed, and merged to develop as they're completed. Jenkins runs CI on every push.
3
Integration testing on develop (Days 15–20)
Once all features are in develop, the full integrated system is tested. Do all the pieces work together? Does workpaper A still work after the Snowflake query change in ticket B?
4
Code freeze — release branch cut (Day 21)
git checkout -b release/2024-07 develop — the release branch is created. No new features after this point. Only release-blocking bug fixes are allowed in. Developers can keep working on next month's features in new feature branches.
5
Release testing (Days 21–27)
The release branch is deployed to the staging environment (not production yet). Final validation. Any bugs found go in as bug fixes on the release branch. These get merged back to develop too.
6
Merge to main + tag (Day 28–31)
Release branch → main. Tag created: git tag -a v2024.07.0 -m "July 2024". Tag pushed to Bitbucket. This is the permanent record of exactly what code is in production.
7
Jenkins deployment fires
Merge to main triggers the Jenkins deployment pipeline. Build → Test → Deploy approval gate → Deploy to production → Control-M job definitions updated → smoke tests run.
8
Close Jira release, notify stakeholders
Mark the Jira release as Released. All attached tickets auto-close. Send release notes to stakeholders. Begin next sprint.

Hotfixes — bypassing the normal release cycle

A hotfix is an emergency fix for a production-breaking bug that can't wait for the next monthly release. It bypasses the normal feature → develop → release → main flow.

# Production is broken. Workpapers are failing to run. Fix needed NOW. git checkout main # start from production code git checkout -b hotfix/AUDIT-170-null-crash # ... fix the bug ... git commit -m "AUDIT-170: fix null loan_id crash in payment extraction" git push origin hotfix/AUDIT-170-null-crash # PR directly to main (emergency — expedited review) # After merge to main: git tag -a v2024.07.1 -m "Hotfix: null crash fix" # ALSO merge the fix back to develop so next release includes it git checkout develop git merge hotfix/AUDIT-170-null-crash git push origin develop
Chapter 10

Environments Explained

An environment is a complete, isolated instance of your application and infrastructure. Most teams have at least three. Understanding them is essential for understanding deployment.

The report development analogy

When you're building a new R Markdown workpaper, you don't test it on live production data with real auditors waiting for output. You test it on a subset of data in a dev environment. Once it looks right, you move it to a staging environment that mirrors production to do final validation. Only then does it go live. Environments in software work exactly the same way.

The standard three environments

Development (dev)
Where developers write and test code. Messy, unstable, frequently broken — and that's fine.
Your local laptop or a shared dev server. Points to dev versions of Snowflake views, dev GREAT microservice instances, test data. Breaking things here has zero business impact.
What gets deployed here: feature branches, constantly. Automatically on every push via CI.
Staging (UAT)
A production-like environment used for final validation before going live. "User Acceptance Testing."
Like running your R workpaper against a full copy of production data but on a server nobody outside the team is watching. Everything should behave exactly like production.
What gets deployed here: the release branch. Business stakeholders validate here before approving production deployment.
Production (prod)
The live system. Real data, real users, real consequences. Changes here require change control.
The scheduled Control-M jobs running against real Snowflake data, producing the actual audit workpapers that go to the business. Any bug here is a real incident.
What gets deployed here: main branch only, during the monthly release window, with an approval gate.

Environment-specific configuration

Code should behave identically across environments — but point at different resources. You shouldn't change code to switch environments. You change configuration. This is done through environment variables.

# In your Python code — never hardcode environment-specific values import os SNOWFLAKE_HOST = os.environ['SNOWFLAKE_HOST'] # dev: dev-snowflake, prod: prod-snowflake GREAT_MS3_URL = os.environ['GREAT_MS3_URL'] # dev: http://ms3-dev, prod: http://ms3-prod LOG_LEVEL = os.environ.get('LOG_LEVEL', 'INFO') # Jenkins sets these environment variables differently per stage: # Dev deploy: SNOWFLAKE_HOST=dev-account.snowflakecomputing.com # Prod deploy: SNOWFLAKE_HOST=prod-account.snowflakecomputing.com

What "promoted to production" means

You'll hear "promote to production" or "promote to prod." This just means: take the artifact (code package) that was successfully tested in staging and deploy that exact same thing to production. You're not rebuilding or redeploying from source — you're moving the already-tested artifact up the environment chain. This is critical because it guarantees what you tested is what you deploy.

Your SSDSP environment

Your team's SSDSP R environment likely maps to one of these tiers. It's probably a staging or production environment where R Markdown workpapers run against real Snowflake data. This is why you connect to it separately from your dev setup — it's isolated for a reason.

You already know how to do this

Workpaper review and code review are identical in purpose. Both ask: does this work product achieve the stated objective, follow team standards, handle edge cases, and leave an understandable trail? The subject matter is different. The review discipline is the same.

Chapter 13

Terminology Glossary

Every technical term you'll hear — with plain-English definitions, your-stack analogies, and context for when you'll encounter each one.

A – G

abstraction
Hiding complex implementation details behind a simple interface
Like a Snowflake view — you query it simply without knowing the complex joins underneath. Good code abstracts complexity so callers don't need to know how something works, just what it does.
API
Application Programming Interface — a defined way for two systems to communicate
Like a standardized form. If you fill in the right fields and submit, you get a predictable response back. Your GREAT microservices expose APIs. External data sources expose APIs your team calls.
REST API: uses HTTP methods (GET, POST, PUT, DELETE) over the web
async / await
A way of writing code that doesn't block waiting for slow operations (like API calls)
Like submitting an R job and continuing to work while it runs instead of staring at the console. Async code sends a request and moves on, handling the response when it arrives.
authentication vs authorization
Authentication = who are you? Authorization = what are you allowed to do?
Authentication is your Snowflake login. Authorization is whether your account has SELECT rights on the LOANS table. You can be authenticated but not authorized.
boilerplate
Standard, repetitive code that appears in many places with little variation
Like the standard headers on every audit workpaper. Necessary, but copy-pasted. Good frameworks reduce boilerplate so developers write less repeated setup code.
dependency
An external library or package your code needs to run
Like R packages — your workpaper depends on dplyr, ggplot2, etc. Python dependencies live in requirements.txt. If a dependency isn't installed, the code fails.
requirements.txt: pandas==2.0.1, snowflake-connector-python==3.1.0
Docker / container
A way to package code and all its dependencies into a portable, isolated box
Like packaging your entire R environment (the R version, all packages, the script) into a zip file that runs identically everywhere. "It works on my machine" goes away because the machine is inside the container.
environment variable
A named value stored outside the code that the code reads at runtime
Like an R Markdown parameter — you pass it in at render time rather than hardcoding it in the script. Keeps secrets out of code and makes the same code run differently in dev vs production.
framework
A pre-built structure that provides standard patterns and tools so you don't start from scratch
Like an R Markdown workpaper template that has standard sections pre-built. A framework gives your code a structure to live inside. FastAPI is a framework for building Python APIs. Your GREAT application is a framework for audit automation.

H – P

idempotent
An operation that produces the same result no matter how many times you run it
Like a SQL INSERT with ON CONFLICT DO NOTHING — running it 5 times produces the same result as running it once. Critical for scheduled jobs that might run twice due to retries.
linting
Automated code style checking — enforces consistent formatting rules across the team
Like spell-check and grammar-check for code. Catches things like inconsistent indentation, unused variables, lines that are too long. Usually runs in the Jenkins pipeline automatically. Python uses flake8 or pylint. R uses lintr.
logging
Writing timestamped records of what a program did to a log file for debugging and auditing
Like an audit trail for code execution. Levels: DEBUG (verbose dev info), INFO (normal operations), WARNING (something unexpected but not broken), ERROR (something failed), CRITICAL (system is down). Production usually logs INFO and above.
microservice
A small, independently deployable service that does one thing well
Your GREAT application is a perfect example. MS1 orchestrates, MS2 triggers Control-M, MS3 is the workpaper repository, MS4 does QA checking. Each is separate — you can deploy MS4 without touching MS1. The alternative (monolith) is one giant application that does everything.
mock / stub
A fake version of an external dependency used in tests so tests run without real connections
Like testing an R workpaper with a hardcoded sample dataset instead of a live Snowflake connection. A mock replaces the Snowflake connector with a fake that returns known test data, so tests run fast and don't depend on the database being up.
orchestration
Coordinating multiple services or processes to work together in sequence
Like your MS1 microservice — it coordinates the whole workflow: trigger MS2, wait for data, trigger workpaper generation, send to MS3. Control-M is also an orchestrator for scheduled jobs.
payload
The data body of an API request or response
Like the body of an email. The envelope (HTTP headers) wraps the payload (the actual data). When your Python code calls a REST API, the payload is usually a JSON object.
{"loan_id": "12345", "as_of_date": "2024-06-30", "status": "delinquent"}
PR (pull request)
A request to merge one Git branch into another, with a built-in review and discussion thread
Workpaper review process, but for code. Preparer opens the PR, reviewer comments and approves, then it's merged (filed). Full trail preserved in Bitbucket.

R – Z

refactoring
Restructuring existing code without changing its behavior — improving readability and maintainability
Like cleaning up a messy SQL query — you rewrite it with CTEs and better formatting, but it still returns the exact same results. No new features, no bug fixes. Pure improvement of the code's internal quality.
regression
When a change breaks something that was working before
You add a new column to a Snowflake query and it breaks the R workpaper that was already working. The workpaper "regressed." Regression tests catch these — they verify old functionality still works after new changes.
REST / RESTful
A standard style for designing web APIs — uses HTTP methods and URLs to represent operations on resources
GET /loans = get all loans. POST /loans = create a loan. PUT /loans/123 = update loan 123. DELETE /loans/123 = delete it. Predictable, readable, widely used. Your GREAT microservices likely expose REST APIs.
schema
The structure and data types of a database table or API payload
Exactly what you know from SQL. Column names, data types, constraints. In Python/API context, a schema also describes the expected shape of a JSON object. A schema change in Snowflake without notifying developers is a common production incident.
technical debt
The accumulated cost of shortcuts, quick fixes, and deferred improvements in code
Like an audit finding you documented but didn't remediate. It's still there, accumulating risk. Technical debt isn't necessarily bad — sometimes a quick solution is appropriate — but it has to be tracked and paid down intentionally, or it slows everything down.
token / JWT
A short-lived credential string used for API authentication instead of a password
Like a visitor badge that expires at end of day. You can't use a visitor badge the next morning — you get a new one. Tokens expire (typically minutes to hours). Your GREAT microservices and Snowflake API connections use tokens. They must be refreshed when they expire.
version control
A system that tracks and manages changes to files over time
Git is the universal standard for code version control. Without it, teams would share code by emailing files. With it, every change is tracked, attributable, reversible, and branchable.
YAML
A human-readable configuration file format — "Yet Another Markup Language"
Like a structured settings file. Used everywhere in DevOps tools for configuration. Jenkins, Kubernetes, GitHub Actions all use YAML. It's whitespace-sensitive — indentation matters, like Python. A misplaced space can break a pipeline config.
key: value
  nested_key: nested_value
  list:
    - item1
    - item2
